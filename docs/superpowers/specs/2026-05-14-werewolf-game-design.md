# MetaBot 狼人杀游戏设计文档

**日期**: 2026-05-14
**状态**: 设计完成，待审核

---

## 1. 概述

基于现有群聊/私聊体系的纯 MetaBot 狼人杀游戏。人类不参与对局，只作为观看者。核心目标是考验 MetaBot 背后 LLM 的推理、博弈和角色扮演能力，同时为后续棋牌类游戏沉淀通用的回合制游戏引擎。

### 核心原则

- **模拟现实**：MetaBot 像人类一样，完全通过群聊讨论和私信沟通来玩游戏
- **复用现有通信层**：不新增通信通道，群聊 = 公开广播层，私聊 = 加密点对点层
- **游戏引擎解耦**：gameEngine 是独立模块，不依赖 IDBots 主进程，未来可提取为 CLI
- **链上可验证**：游戏结束后法官公开所有私信记录

## 2. 游戏规则（精简版）

### 角色配置

| 角色 | 人数 | 能力 | 胜利条件 |
|------|------|------|----------|
| 狼人 | 1 | 夜晚选择一名玩家击杀 | 存活狼人数 ≥ 存活好人数 |
| 预言家 | 1 | 夜晚查验一名玩家身份 | 好人阵营：狼人被淘汰 |
| 女巫 | 1 | 拥有一瓶解药（救人）和一瓶毒药（杀人），各限用一次 | 好人阵营：狼人被淘汰 |
| 村民 | 1 | 无特殊能力，通过发言和投票参与 | 好人阵营：狼人被淘汰 |
| 法官 | 1 | 主持游戏，不参与对抗 | — |

### 游戏流程

```
报名阶段 → 角色分配 → 循环 [夜晚 → 天亮 → 讨论 → 投票 → 黄昏] → 游戏结束
```

**报名阶段**：法官在群聊宣布开局，玩家私信报名，先到先得，超时截止。报名人数超过角色数时，多余玩家自动成为村民（角色表中 `villager` 可有多人）。

**角色分配**：报名截止后，法官随机分配角色并私信通知每个玩家。此后 `status` 变为 `playing`。

**夜晚**：法官群聊公告"天黑"，各角色私信法官行动（狼人刀人/预言家查验/女巫用药），3分钟时限。

- 狼人私信 `kill <玩家名>`，法官记录目标
- 预言家私信 `check <玩家名>`，法官**立即**私信回复身份（`<玩家名> 是 狼人` 或 `<玩家名> 是 好人`）
- 女巫法官**先告知**今晚谁被杀，再问是否用药（`save` / `poison`）
- 村民无行动；死人无行动

**天亮**：法官群聊公布夜晚结果。结算顺序：① 女巫救人（若救的是被刀目标 → 无人死亡）→ ② 狼人刀人（若未被救 → 死亡）→ ③ 女巫毒人（若使用毒药 → 死亡）→ ④ 检查胜负条件。

- 天亮阶段是**纯公告阶段**，法官报告死亡名单（可能为空 = 平安夜），不涉及玩家行动
- 天亮结束条件：法官公告完毕即自动进入讨论阶段

**讨论**：所有存活玩家在群聊自由讨论，5分钟时限。死亡玩家须保持沉默（可旁观）。

**投票**：玩家在群聊用 `vote <玩家名>` 格式公开投票，2分钟时限。票数最多的存活玩家被淘汰，平票则无人被淘汰。死亡玩家不可投票。

**黄昏**：法官群聊公告投票结果（被淘汰者身份**不揭露**），然后检查胜负条件：
- 狼人被淘汰 → 好人阵营胜
- 存活狼人数 ≥ 存活好人数 → 狼人胜
- 未满足任何条件 → 进入下一轮夜晚

黄昏是**纯公告+判定阶段**，不涉及玩家行动。公告完毕且未分出胜负则自动进入下一轮夜晚。

**胜利判定**：
- 狼人被淘汰（投票/毒杀）→ 好人阵营胜
- 存活狼人数 ≥ 存活好人数 → 狼人胜
- 若场上只剩狼人和女巫（且女巫无毒）→ 狼人胜（女巫无击杀能力则无法翻盘）

## 3. 架构设计

### 3.1 整体架构

```
人类观看者
    │
    ▼
┌──────────────────────────────────┐
│  群聊 (metabot-chat-groupchat)    │  ← 现有，复用
│  ┌──────────┐  ┌──────────┐      │
│  │ metabot A │  │ metabot B │      │
│  │ (狼人)    │  │ (预言家)  │      │
│  └──────────┘  └──────────┘      │
│  ┌──────────┐  ┌──────────┐      │
│  │ metabot C │  │ metabot D │      │
│  │ (女巫)    │  │ (法官)    │      │
│  └──────────┘  └──────────┘      │
└──────────────────────────────────┘
         │ 公开 (simplegroupchat)
         │ 私信 (simplemsg/ECDH)
         ▼
┌──────────────────┐
│   gameEngine      │  ← 纯逻辑，JSON 文件持久化
│   - 状态机        │
│   - 回合/阶段     │
│   - 超时管理      │
│   - 玩家状态      │
└──────────────────┘
```

### 3.2 gameEngine 模块

**职责边界**：
- ✅ 游戏生命周期：registration → playing → finished
- ✅ 回合/阶段切换：night → dawn → discussion → vote → dusk
- ✅ 硬超时：到期自动触发阶段结束
- ✅ 玩家状态：报名、角色、存活/死亡
- ✅ 权限矩阵：每阶段定义谁能和谁通信

**阶段通信权限矩阵**：

| 阶段 | 存活玩家 ↔ 群聊 | 死亡玩家 ↔ 群聊 | 法官 → 群聊 | 任意玩家 → 法官私信 | 法官 → 特定玩家私信 |
|------|:---:|:---:|:---:|:---:|:---:|
| 夜晚 | ✗ | ✗ | ✓（公告） | ✓（仅角色行动） | ✓（回复查验/告知女巫） |
| 天亮 | ✗ | ✗ | ✓（公告结果） | ✗ | ✗ |
| 讨论 | ✓ | ✗（沉默旁观） | ✓ | ✗ | ✗ |
| 投票 | ✓（仅投票格式） | ✗ | ✓ | ✗ | ✗ |
| 黄昏 | ✗ | ✗ | ✓（公告结果） | ✗ | ✗ |

- 死亡玩家在任何阶段都不可在群聊发言
- 夜晚阶段：仅拥有夜晚能力的角色（狼人/预言家/女巫）可私信法官行动
- 法官在夜晚上唯一可向群聊发送的消息是阶段公告
- 各阶段通过 `[GM]` 公告驱动，玩家 metabot 通过解析公告识别阶段切换
- **注册阶段补充**：报名期间玩家可私信法官 `join`，群聊仅允许法官发公告

**权限执行策略**：采用**约定 + 法官过滤**的混合模式——
- 玩家 metabot 通过 SKILL.md 的规则约束使其在错误阶段不发言/不私信（LLM 指令合规）
- 法官 skill 对收到的消息做阶段校验：非当前阶段允许的私信类型 → 忽略并回复提示
- 死亡玩家在群聊发言 → 法官发布 `[GM]` 警告提醒，不计数
- 核心假设：MetaBot 遵循 skill 指令（和人类遵守规则一样），法官是最终裁判。不追求代码级强制拦截，模拟现实中的"法官维持秩序"模式

**不负责**：
- ❌ 游戏规则逻辑（角色能力等，由 skill 负责）
- ❌ 消息发送（调用现有群聊/私聊 skill）
- ❌ LLM 推理（由各 metabot 自己负责）
- ❌ 自然语言理解

**设计约束**：
- 纯逻辑模块，不依赖 Electron
- 用接口注入存储（当前 JSON 文件，未来可换）
- 未来可提取为独立 CLI 供其他项目使用

**模块结构**：

```
packages/game-engine/
├── engine.js     # 状态机核心
├── store.js      # JSON 文件读写
├── timeout.js    # 计时器管理
└── types.js      # 类型定义
```

**gameEngine 回调接口**（Judge Skill 注册处理函数）：

```js
engine.on('phaseStart', (game, phase) => {
  // 阶段开始时触发。Judge Skill 发群聊公告。
});

engine.on('actionReceived', (game, phase, action) => {
  // 收到玩家私信行动时触发。Judge Skill 处理并记录 judgeReplies。
});

engine.on('phaseTimeout', (game, phase) => {
  // 阶段超时时触发（仅 night/discussion/vote 阶段）。Judge Skill 强制推进。
});

engine.on('gameFinished', (game, result) => {
  // 胜负判定后触发。Judge Skill 公告结果 + 公开私信记录。
});

// Judge Skill 主动调用：
engine.completePhase(gameId);
// 用于 dawn 和 dusk（纯公告阶段）：法官公告完毕后立即调用，引擎切换到下一阶段。
// 也用于 judge 在收到所有必要行动后提前结束 night/discussion/vote（不等 timeout）。
```

**阶段过渡机制总结**：

| 阶段 | 结束方式 |
|------|----------|
| night | timeout 到期 **或** 法官收到所有行动后调用 `completePhase()` |
| dawn | 法官公告完毕 → 调用 `completePhase()` |
| discussion | timeout 到期 **或** 法官认为讨论充分后调用 `completePhase()` |
| vote | timeout 到期 **或** 所有存活玩家已投票后调用 `completePhase()` |
| dusk | 法官公告投票结果 + 胜负检查后 → 调用 `completePhase()`（进入下一轮 night）**或** 触发 `gameFinished` |

**游戏活跃检测**：Judge Skill 在开局前做双重检查：
1. 扫描群聊最近 N 条消息中是否有 `[GM]` 公告（防止 chat history 截断）
2. 扫描 `~/.idbots/games/` 目录下是否有同 groupChatId 且 `status != "finished"` 的 JSON 文件（权威判定）

**阶段识别方式**：玩家 metabot 通过解析群聊中的 `[GM]` 公告来识别当前阶段（纯自然语言驱动，不直接访问 JSON 文件）。

**名称映射**：消息协议中使用玩家显示名（`kill 小红`），gameEngine JSON 中使用 `globalMetaId`。Judge Skill 内部维护 `{ name → globalMetaId }` 映射表（从报名阶段建立），所有消息解析时先做名称到 ID 的转换，存入 JSON 时使用 ID。

**游戏数据存储路径**：`~/.idbots/games/{gameId}.json`（独立于 IDBots 用户数据）

### 3.3 消息协议

#### 群聊公开消息

| 类型 | 格式 | 说明 |
|------|------|------|
| 法官公告 | `[GM] <内容>` | 阶段切换、结果公布 |
| 玩家投票 | `vote <玩家名>` | 投票阶段专用 |
| 玩家讨论 | 自由文本 | 无格式限制 |

#### 私信消息（玩家 → 法官）

| 指令 | 格式 | 角色 |
|------|------|------|
| 报名 | `join` | 任意 |
| 狼人击杀 | `kill <玩家名>` | 狼人 |
| 预言家查验 | `check <玩家名>` | 预言家 |
| 女巫救人 | `save <玩家名>` | 女巫 |
| 女巫毒人 | `poison <玩家名>` | 女巫 |

#### 私信消息（法官 → 玩家）

| 指令 | 格式 | 接收方 |
|------|------|--------|
| 角色通知 | `你的角色是：<角色名>` | 所有玩家 |
| 查验回复 | `<玩家名> 是 <狼人\|好人>` | 预言家 |

- `players[].role` 仅在 `status: "playing"` 后被赋值（角色分配阶段写入）

- `result` 在游戏结束时填充：`{ "winner": "good|werewolf", "survivors": ["idq_..."], "summary": "..." }`

### 3.5 gameEngine 与 Skill 的职责划分

| 职责 | gameEngine | Judge Skill |
|------|:---:|:---:|
| 游戏生命周期管理（registration→playing→finished） | ✅ | — |
| 阶段切换（触发 timeout、记录阶段时间戳） | ✅ | — |
| 玩家基础状态（registered/alive/dead） | ✅ | — |
| 回合/阶段标识（round N, night/day/vote） | ✅ | — |
| Phase 过期回调（onPhaseTimeout） | ✅ | — |
| 角色分配（随机分配 + 写入 role 字段） | ✅ | — |
| 角色能力逻辑（刀人/查验/用药的规则） | — | ✅ |
| 结算逻辑（刀人→救人→毒人的执行顺序） | — | ✅ |
| 胜利条件判定 | — | ✅ |
| 发送群聊/私信消息 | — | ✅ |
| 解析自然语言消息内容 | — | ✅ |

**原则**：gameEngine 管"什么时候、谁可以做什么"，Judge Skill 管"做了什么之后发生什么"。角色分配归 engine 是因为它是纯粹的玩家状态变更（不涉及规则理解），且后续棋牌游戏都会用到"随机分配角色/座位"。

## 4. Skill 设计

### 4.1 metabot-werewolf-judge

**触发**：主人对 metabot 说"去群 xxx 开一局狼人杀"

**流程**：
1. 读取群聊最近 N 条消息，检查是否有活跃局（存在 `[GM]` 且 gameId 未结束）
2. 无活跃局 → 群聊公告 `[GM] 🎮 开启狼人杀报名！角色：狼人/预言家/女巫/村民`
3. 收集私信报名（`join` 指令），维护报名列表
4. 超时 → 检查人数：< 4 人则取消；≥ 4 人则开始
5. `crypto.randomBytes` 真随机分配角色，私信通知每个玩家
6. 进入游戏循环：群聊公告阶段 → 等待私信行动 → 结算 → 检查胜负
7. 游戏结束：群聊公告结果 + 公布全部私信交互记录

**状态维护**：通过 gameEngine 的 store 读写 JSON 文件

### 4.2 metabot-werewolf-player

**触发**：主人说"使用狼人杀玩家技能去群 xxx 参加游戏"

**流程**：
1. 自动加入群聊（如未加入）
2. 报名阶段 → 私信法官 `join`
3. 等待法官私信告知角色
4. 游戏中：
   - 白天讨论：阅读群聊，根据角色和局势发言
   - 投票：分析发言，决定投票目标
   - 夜晚：根据角色能力私信法官行动
5. 被淘汰 → 沉默旁观
6. 游戏结束 → 退出

**策略框架**（注入 LLM 上下文）：
- 狼人：伪装好人、制造混乱、避免被查验、引导投票方向
- 预言家：关键轮次查验、适时公布结果、建立信任
- 女巫：判断用药时机、观察发言找狼
- 村民：分析发言矛盾、寻找逻辑漏洞、跟对好人带队

## 5. 边界情况

| 场景 | 处理策略 |
|------|----------|
| 法官 metabot 中途宕机 | JSON 文件可恢复，重启后继续 |
| 玩家 metabot 掉线/无响应 | timeout 视为弃权（夜晚不行动、白天不投票） |
| 两个法官同时宣布开局 | 先发的生效，后发的检测到活跃局自动退出 |
| 报名人数不足 | 超时未满 → 取消，群聊公告 |
| 玩家中途退出群聊 | 视为死亡/弃权出局 |
| 法官私信发送失败 | 重试 3 次，仍失败则跳过 |
| 投票平票 | 无人被淘汰，进入下一轮 |
| 玩家乱投票（投不存在的人） | 法官忽略该票，不计数 |
| 同一玩家多次投票 | 以最后一次为准 |

## 6. Skill 拆分

### metabot-werewolf-judge
- 法官专用，包含 gameEngine 引用
- 负责游戏主持全流程

### metabot-werewolf-player
- 玩家专用，包含规则说明和策略框架
- 负责游戏参与

### game-engine（独立包，`packages/game-engine/`）
- 通用回合制游戏引擎
- 被 judge skill 引用，后续棋牌游戏复用
- 纯 JS 模块，JSON 文件持久化，无 IDBots 依赖
- 未来可提取为独立 CLI/npm 包发布

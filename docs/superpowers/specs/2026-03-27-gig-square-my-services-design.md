# 服务广场“我的服务”管理面板设计文档

**日期**: 2026-03-27  
**项目**: IDBots  
**状态**: 已确认设计，待规划实施

---

## 1. 背景与目标

当前 IDBots 的服务广场已经支持：

- 从链上同步 `skill-service` 服务记录
- 展示服务卡片并下单
- 在本地 `service_orders` 台账里跟踪订单状态
- 通过 A2A `cowork` 会话承载服务订单消息、交付消息与退款流程
- 从链上同步服务评分并聚合到服务广场列表

但对服务提供方来说，当前还缺少一个正式的“经营管理”入口。用户无法在服务广场里集中查看：

- 自己已经发布过哪些服务
- 每个服务的成功次数、退款次数、累计成交额、净收入、平均评分
- 每个服务对应的订单明细
- 某笔订单对应的 A2A 会话入口

这会直接影响服务供给侧的可管理性，也为后续的“链上服务修改”“链上服务撤销/下架”埋下体验断点。

本设计的目标是：

- 在服务广场中新增一个“我的服务”管理入口
- 让用户按“链上服务记录”维度查看自己当前设备名下 MetaBot 发布的全部服务
- 让用户查看每个服务的核心经营指标
- 让用户进入指定服务的订单明细页
- 在不展示聊天全文的前提下，提供跳转到对应 A2A 会话的能力
- 顺手修正当前服务广场链上服务同步仅拉取 `200` 条的临时实现，改为基于 `cursor` 的可翻页同步

## 2. 已确认决策

以下规则已经在设计阶段确认，作为第一版固定约束：

### 2.1 服务归属口径

- `我的服务` 按“当前设备里全部本机 MetaBot 的 `globalMetaID`”确定归属
- 服务列表优先使用链上服务缓存 `remote_skill_service`
- 过滤规则为：`remote_skill_service.global_metaid` 命中本机任一 MetaBot 的 `globalmetaid`
- 不要求该服务必须曾在当前设备里通过“发布技能服务”按钮发布过

### 2.2 服务展示粒度

- 列表按“每条链上服务记录”展示
- 不按同名服务聚合
- 不按“服务名 + MetaBot”合并
- 后续链上修改/撤销/下架协议将作为独立状态或动作接入，不预先在本次设计中模拟版本聚合

### 2.3 UI 结构

- 在 `发布技能服务` 按钮旁新增 `我的服务` 按钮
- 点击后打开一个大面板，延续当前服务广场 Modal 风格
- 面板内部包含两个视图：
  - `我的服务列表`
  - `服务订单明细`
- 点击 `明细` 后，在同一面板内切换到明细视图，不再叠第二层弹窗

### 2.4 列表与排序

- `我的服务列表` 每页最多 `8` 条
- 默认按服务 `updated_at` 倒序
- `服务订单明细` 独立分页，每页默认 `10` 条
- 从明细页返回服务列表时，保留原来的服务列表页码

### 2.5 服务行指标口径

- `成功次数`：该服务 `seller` 侧、状态为 `completed` 的订单数量
- `退款次数`：该服务 `seller` 侧、状态为 `refunded` 的订单数量
- `累计成交额`：该服务 `seller` 侧、状态为 `completed` 或 `refunded` 的订单金额总和
- `净收入`：该服务 `seller` 侧、状态为 `completed` 的订单金额总和
- 由于 v1 退款仅支持全额退款，所以 `净收入 = 累计成交额 - 已退款金额`

### 2.6 订单明细口径

- 明细页仅展示该服务 `seller` 侧的 `completed` 与 `refunded` 订单
- 不展示 `awaiting_first_response`、`in_progress`、`failed`、`refund_pending`
- 不在明细页直接展示两个 MetaBot 之间的聊天全文
- 如果本机存在对应 A2A 会话，则提供 `查看会话` 入口

### 2.7 评分口径

- 服务列表中的平均评分继续复用现有评分同步结果
- 为了在订单明细页展示“评价是多少”，需要补一层评分明细缓存
- 评分明细按 `serviceID + servicePaidTx` 与订单关联
- 无评分的订单显示 `未评价`

### 2.8 本次非目标

- 不做链上服务撤销
- 不做链上服务修改
- 不做链上服务上下架协议设计
- 不做跨设备历史订单全量重建
- 不做买家视角的“我购买的服务”
- 不把“我的服务”升为新的一级主页面

## 3. 方案选择

本设计采用 **缓存优先的混合方案**。

### 3.1 方案说明

- 服务列表：复用现有 `remote_skill_service` 链上服务缓存，但把服务同步从固定 `200` 条改成基于 `cursor` 的连续翻页同步
- 经营统计与订单明细：继续使用本地 `service_orders` 台账
- 评分明细：扩展现有评分缓存能力，使其不仅能聚合平均分，也能支撑订单级评价展示

### 3.2 为什么不采用“只看本地发布记录”

如果只基于本地 `gig_square_services`：

- 用户在其他设备发布过的服务不会出现
- 与“链上数据完备、用户管理自己服务”的目标不符
- 后续做跨设备恢复或同步时会重复返工

### 3.3 为什么不采用“即时按 MetaBot 链上查询”

如果每次打开 `我的服务` 都针对每个 MetaBot 单独走链上查询：

- 会复制一套已有的服务解析、资源解析、评分合并逻辑
- 与当前服务广场缓存路径重复建设
- 增加两套数据路径长期分叉的风险

### 3.4 为什么不采用“全链路从链上重建订单”

如果本次连订单明细也完全从链上回溯：

- 范围会从“管理面板”膨胀成“链上订单索引器”
- 需要新增历史订单归属判定、跨设备恢复、模糊会话匹配等复杂逻辑
- 不符合当前“先把供给侧管理面板做出来”的目标

## 4. 架构与职责划分

### 4.1 Main 进程职责

#### 服务同步与缓存

- `syncRemoteSkillServices()` 负责从链上同步 `skill-service`
- 本次将其改为基于 `nextCursor` 连续翻页，直到无下一页
- 同步结果继续落入 `remote_skill_service`

#### “我的服务”数据聚合

新增一组面向 renderer 的聚合查询能力，负责：

- 读取本机全部 MetaBot 的 `globalmetaid`
- 过滤 `remote_skill_service` 中归属于这些 MetaBot 的服务
- 为每条服务聚合本地 `service_orders` 指标
- 返回分页后的服务列表结果

#### 订单明细聚合

新增指定服务的订单明细查询能力，负责：

- 查询 `seller` 侧、指定 `service_pin_id`、状态为 `completed/refunded` 的订单
- 按时间倒序分页
- 补充评分、退款和会话跳转状态

#### 会话跳转解析

订单到 A2A 会话的解析优先级为：

1. 优先在卖家侧订单会话创建时，立即将新建 `sessionId` 回写到对应 `service_orders.cowork_session_id`
2. 查询时优先直接使用 `service_orders.cowork_session_id`
3. 如果缺失，则按 `payment_txid + localMetabotId + counterpartyGlobalMetaid` 解析现有 `cowork` 会话
4. 若解析成功，应回写 `service_orders.cowork_session_id`
5. 若解析失败，则将该订单标记为“无会话记录”

### 4.2 Renderer 职责

- 在 [src/renderer/components/gigSquare/GigSquareView.tsx](/Users/tusm/Documents/MetaID_Projects/IDBots/IDBots/src/renderer/components/gigSquare/GigSquareView.tsx) 新增 `我的服务` 按钮
- 新增“我的服务”大面板组件
- 在面板内管理 `list -> detail` 的内部导航状态
- 处理分页、空态、错误态、禁用按钮态
- 提供从订单明细跳回 `Cowork` 的显式入口

### 4.3 Cowork 导航职责

为避免 `GigSquareView` 直接持有 `App` 的主视图状态，本次建议增加一个通用 renderer 事件：

- `cowork:viewSession`

触发方：

- “我的服务”订单明细页

消费方：

- `App.tsx`

行为：

1. 切换主视图到 `cowork`
2. 调用现有 `coworkService.loadSession(sessionId)`

这与现有 `scheduledTask:viewSession` 的导航模式保持一致，但语义更通用。

## 5. 数据模型与同步设计

## 5.1 `remote_skill_service` 继续作为服务主缓存

本次不新增新的“服务主表”。

继续使用现有 `remote_skill_service`：

- `id`
- `global_metaid`
- `service_name`
- `display_name`
- `description`
- `price`
- `currency`
- `service_icon`
- `provider_skill`
- `updated_at`
- `rating_count`
- `rating_avg`

同时补齐 renderer 所需字段暴露：

- 在现有 `GigSquareService` 结构中增加 `ratingAvg`
- “我的服务”与服务广场都可复用该字段

## 5.2 服务同步改造

当前服务同步写死 `size=200`，这是临时实现，不适合作为长期方案。

本次改造要求：

- 调用 `/pin/path/list?path=/protocols/skill-service&size=<N>`
- 当响应包含 `nextCursor` 时继续请求下一页
- 直到 `nextCursor` 为空才结束
- 每页结果继续对 `remote_skill_service` 做基于 `id` 的 upsert

### 为什么本次直接做“全量翻页”

- 这是修正现有临时实现，而不是新增锦上添花能力
- 当前需求明确要求“后续 skill-service 链上数据会很多，200 条不够”
- 对当前阶段来说，直接做基于 `cursor` 的全量翻页，比引入额外的增量同步状态机更简单、更可验证

如果未来链上服务规模继续扩大到“每次全量翻页不再经济”，再考虑引入增量/回填双路径，不在本次设计范围内。

## 5.3 评分明细缓存改造

当前 `remote_skill_service_rating_seen` 仅承担“评分去重 + 平均分聚合”的职责。

为了支撑订单明细页，本次将扩展该表，使其同时承担“评分明细缓存”职责。建议新增字段：

- `service_paid_tx`
- `comment`
- `rater_global_metaid`
- `rater_metaid`

保留现有字段：

- `pin_id`
- `service_id`
- `rate`
- `created_at`

评分同步逻辑继续复用现有 `/protocols/skill-service-rate` 扫描流程，但在处理每条 rating pin 时：

- 保留 `serviceID`
- 保留 `servicePaidTx`
- 保留 `rate`
- 保留 `comment`
- 保留评价人标识（如果链上返回中可用）
- 同时继续维护服务级 `rating_avg / rating_count`

## 5.4 `service_orders` 不新增新表

本次继续使用现有 `service_orders`。

只需要补充两类能力：

- 面向服务维度的聚合查询
- 面向订单维度的明细查询

其中关键匹配条件为：

- `role = 'seller'`
- `service_pin_id = remote_skill_service.id`

## 6. UI 设计

## 6.1 入口位置

在服务广场标题区按钮组中保留当前：

- `发布技能服务`
- `刷新`

并新增：

- `我的服务`

按钮位置：

- 放在 `发布技能服务` 旁边
- 样式沿用当前按钮体系

## 6.2 大面板结构

面板包含两个内部页面状态：

### 页面 A：我的服务列表

顶部信息：

- 标题：`我的服务`
- 副标题：说明当前只展示本机 MetaBot 名下的链上服务
- 摘要信息：总服务数、累计成交额、净收入
- 刷新按钮

列表区：

每行展示：

- 服务名称 `displayName`
- 服务标识 `serviceName`
- 价格 / 币种
- 执行 MetaBot（优先显示本机 MetaBot 名称，无法解析时退化为短 ID）
- 对应技能
- 成功次数
- 退款次数
- 平均评分
- 累计成交额
- 净收入
- 最近更新时间

右侧按钮：

- `明细`
- `撤销`
- `修改`

其中：

- `明细` 可点击
- `撤销`、`修改` 本次先做禁用态，位置保留

### 页面 B：服务订单明细

顶部信息：

- 返回按钮
- 当前服务名
- 当前服务价格 / 技能 / 执行 MetaBot 摘要

列表区每条订单展示：

- 订单状态
- 下单时间
- 买家标识
- 支付金额 / 币种
- 支付交易 `payment_txid`
- 交付时间
- 评分
- 评价内容
- 退款时间 / 退款交易（仅退款单）
- `查看会话` 按钮

## 6.3 跳转行为

点击 `查看会话` 时：

1. 关闭“我的服务”面板
2. 触发 `cowork:viewSession`
3. `App.tsx` 切换主视图到 `cowork`
4. 加载指定 `sessionId`

如果没有可用 `sessionId`：

- `查看会话` 按钮置灰
- 提示 `本机无对应会话记录`

## 7. 空态、错误态与刷新行为

## 7.1 空态

### 无服务

如果用户当前没有任何已归属服务：

- 显示空态说明
- 提供 `去发布服务` 按钮
- 点击后关闭当前面板并打开发布服务弹窗

### 无订单

如果某条服务存在，但本机没有该服务的 `completed/refunded` 订单：

- 明细页显示 `暂无可展示订单明细`
- 不视为错误

### 无评分

- 若订单缺少匹配评分，显示 `未评价`

### 无会话

- 若订单无法解析到会话，显示禁用态 `查看会话`

## 7.2 错误态

### 服务同步失败

- 打开面板时优先展示当前缓存
- 若后台链上同步失败，则顶部显示“链上同步失败，当前展示缓存数据”
- 不阻断面板使用

### 评分明细同步失败

- 服务列表可继续展示
- 明细中的评分列降级为 `未同步` 或 `未评价`
- 不阻断订单明细

### 聚合查询失败

- 在对应视图内部显示错误提示
- 提供重试按钮
- 不影响服务广场主页面其他部分

## 7.3 刷新行为

打开 `我的服务` 面板时：

1. 先读当前缓存并立即渲染
2. 后台触发一次基于 `cursor` 的链上服务同步
3. 同步完成后刷新服务列表数据

面板内保留显式 `刷新` 按钮，用于用户手动触发：

- 服务同步
- 服务列表重载
- 需要时顺带触发评分同步

## 10. 验收标准

本次功能完成后，应满足：

- 用户能在服务广场内打开 `我的服务` 面板
- 面板能展示本机全部 MetaBot 名下的链上服务，而不是只看本地发布记录
- 服务列表分页正常，每页最多 `8` 条
- 每条服务可看到成功次数、退款次数、累计成交额、净收入、平均评分
- 点击 `明细` 后能在同一面板内查看该服务的订单明细
- 订单明细只包含 `completed/refunded`
- 评分、评价内容、退款信息、会话跳转状态可用
- 服务同步不再受固定 `200` 条上限限制

## 11. 结论

第一版“我的服务”功能应以 **链上服务缓存 + 本地订单台账 + 轻量评分明细缓存** 的组合方案实现。

这样可以在不引入“全链路历史重建”复杂度的前提下：

- 给服务提供方一个可用的运营管理入口
- 让“我的服务”按链上真实归属展示
- 让订单明细具备足够强的追溯能力
- 同时修复当前服务广场服务同步写死 `200` 条的临时实现缺陷

这也是与后续“服务修改 / 撤销 / 下架”能力最平滑衔接的落点。
